# EZLove 易挂念 — OpenSpec v1.0

## 1. 项目愿景

> **让关心自然流动，让牵挂被看见。**

独居老人最大的困境不是缺少监控，而是缺少牵挂。EZLove 反转守护范式：不监控老人，而是让子女的日常关心自然流向老人。老人感受到的是"孩子想着我"，系统感知到的是"老人今天在线"。

## 2. 命题对齐

| 命题要求 | EZLove 方案 |
|---------|-----------|
| 不用摄像头 | 纯软件方案，无硬件 |
| 老人端零学习成本 | 老人只需点微信里的卡片消息 |
| 守护≠监控 | 老人感知到的是家人分享，不是被监控 |
| 不做医疗诊断 | 仅做在线状态检测，不涉及健康数据 |
| 尊重情感和隐私 | 主动权在老人手中，选择看不看；不采集任何生物数据 |
| 社区协同响应 | 异常时可一键通知社区网格员/物业 |
| 可运行 demo | uni-app 小程序 + FastAPI 后端，完整可运行 |
| 可用 AI 工具 | AI 辅助生成分享内容，降低子女使用门槛 |

## 3. 用户角色

### 3.1 子女/亲属（Family）
- **画像**：25-50 岁，工作忙碌，不在老人身边，想关心但常常忘记或不知道发什么
- **痛点**：每天主动打电话有心理负担，怕打扰；想确认老人安好但没有轻量方式
- **核心需求**：用最低成本表达关心，同时确认老人每天都"在线"

### 3.2 独居老人（Elder）
- **画像**：60-85 岁，会用微信看消息，可能不会主动发消息
- **痛点**：孤独感，觉得孩子不想着自己；不愿意被当成需要"看管"的对象
- **核心需求**：感受到被牵挂，看到孩子的日常，保持尊严

### 3.3 社区联系人（Community，Phase 2）
- **画像**：社区网格员、物业管家、邻居志愿者
- **痛点**：不知道哪个老人需要上门关注
- **核心需求**：在老人可能出现异常时收到精准通知

## 4. 核心用户故事

### P0 — MVP 必做

#### US-01：子女发送牵挂
```
作为子女，我想快速分享一条日常（照片/文字/语音）给我的父母，
这样他们能感受到我的牵挂。
```
**验收标准**：
- 子女打开小程序 → 选择图片或输入文字 → 一键发送
- 支持图片 + 文字组合
- 发送后可在"发送记录"中查看
- 发送成功后生成微信转发卡片，可转发给老人或家庭群

#### US-02：老人查看牵挂
```
作为老人，我收到孩子发来的微信消息卡片，
点开就能看到孩子分享的内容，不需要学任何新东西。
```
**验收标准**：
- 老人在微信聊天中看到小程序卡片 → 点击打开 → 直接展示内容
- 页面大字体、高对比度、极简布局
- 无需登录、无需注册（通过小程序静默授权获取 openid）
- 支持点击"已看到"按钮（大按钮，可选的主动反馈）

#### US-03：已读状态追踪
```
作为系统，当老人打开查看了分享内容时，
我需要记录查看时间，作为"今日在线"的信号。
```
**验收标准**：
- 老人打开页面时自动记录 view 事件（openid + 时间 + 内容 ID）
- 子女端可看到每条内容的已读/未读状态
- 子女端首页显示"父亲/母亲今日状态：已读 ✓ / 未读"

#### US-04：未读异常提醒
```
作为子女，当我发送的内容超过设定时间未被查看时，
我希望收到提醒，好让我主动联系确认老人状况。
```
**验收标准**：
- 子女可设置"提醒阈值"（默认：当日 20:00 前未读则提醒）
- 超时后通过小程序订阅消息通知子女
- 提醒消息包含老人姓名 + 最近一次在线时间

#### US-05：AI 辅助生成内容
```
作为子女，当我忙碌或不知道发什么时，
AI 可以帮我生成适合的内容建议，我确认后一键发送。
```
**验收标准**：
- 子女点击"AI 帮我写" → 系统根据上下文生成 2-3 条内容建议
- 内容类型：天气问候、养生提示、节气问候、回忆触发（"今天是您最爱的桂花开的时候"）
- 子女可编辑修改后发送
- AI 生成内容标注来源（但老人端不显示是否 AI 生成）

#### US-06：家庭绑定
```
作为子女，我需要绑定我的父母，建立牵挂关系。
```
**验收标准**：
- 子女生成"绑定邀请码"或"绑定二维码"
- 老人扫码或输入邀请码完成绑定（可由子女帮操作）
- 一个老人可被多个子女绑定
- 一个子女可绑定多个老人
- 绑定关系支持备注（如"爸爸"、"妈妈"、"外婆"）

### P1 — 第二阶段

#### US-07：社区联系人通知
```
作为子女，当我无法联系到老人时，
我可以一键通知社区联系人帮忙上门查看。
```
**验收标准**：
- 子女可预设社区联系人（手机号 + 姓名 + 关系）
- 未读告警页面提供"请求社区协助"按钮
- 点击后向社区联系人发送短信/小程序通知

#### US-08：老人日活跃度看板
```
作为子女，我想看到老人最近一周的"在线"模式，
了解老人的日常活跃规律。
```
**验收标准**：
- 展示最近 7/30 天的"已读打卡日历"
- 标注每天首次查看时间
- 异常模式提示（如"最近 3 天查看时间明显推迟"）

#### US-09：老人主动回应
```
作为老人，看到孩子分享的内容后，
我想用最简单的方式回应（表情/语音）。
```
**验收标准**：
- 老人查看内容后可选择发送预设表情（竖大拇指、爱心、笑脸等大图标）
- 支持长按录制语音回应（≤15 秒）
- 子女端可查看老人的回应

### P2 — 远期

- US-10：多子女协作（分配每日"值班"发送任务）
- US-11：老人端信息流（汇聚所有子女的分享，像朋友圈一样浏览）
- US-12：节日/生日特别关怀模板
- US-13：智能频次调整（根据老人活跃度自动调整提醒策略）

## 8. AI 内容生成策略

### 8.1 生成逻辑

AI 内容生成服务接收以下上下文，生成个性化的分享建议：

**输入上下文**：
- 当前时间、天气、节气、节日
- 老人的偏好标签（子女填写：喜欢花草/烹饪/戏曲/新闻等）
- 最近发送的内容（避免重复）
- 老人所在城市（天气/本地信息关联）

**输出格式**：
每次返回 2-3 条建议，每条包含：
- 文字内容（≤100 字，温暖、口语化）
- 配图建议（可选，从素材库匹配或提示子女自拍）
- 内容类型标签（天气问候/养生/节气/回忆/日常）

**示例生成**：
```
☀️ "妈，杭州今天 32 度，记得多喝水。我今天中午吃了你教我做的红烧肉，味道越来越像你做的了！"

🌿 "爸，今天芒种，你阳台上的薄荷该浇水了吧？我这边的绿萝长得可好了，回头拍给你看。"

📸 "妈，今天加班路上看到一家新开的糕点店，买了你最爱的桂花糕，下次回来给你带！"
```

### 8.2 技术实现

- 调用 Claude API / 通义千问 API 生成内容
- Prompt 模板存储在后端，可动态更新
- 使用 Redis 缓存已生成内容，避免重复
- 生成内容经过安全过滤后返回

## 9. 告警系统设计

### 9.1 检测机制

```
定时任务（APScheduler，每小时执行一次）
  │
  ├─ 遍历所有 active 的 care_relations
  │
  ├─ 检查每对关系中，今日是否有 care_moment
  │   └─ 无 → 跳过（子女今天没发，不触发告警）
  │
  ├─ 有 care_moment → 检查是否有对应的 view_event
  │   ├─ 已读 → 正常，更新 last_active_at
  │   └─ 未读 → 检查是否超过 alert_threshold
  │       ├─ 未超过 → 等待
  │       └─ 已超过 → 触发告警
  │
  └─ 告警分级：
      ├─ info：超过阈值但在24h内 → 提醒子女"今日内容尚未被查看"
      ├─ warning：超过24h未读 → "已超过24小时未查看，建议联系确认"
      └─ urgent：连续2天+ 未读 → "连续X天无活跃，请尽快确认老人状况"
```

### 9.2 通知渠道

- **小程序订阅消息**（主要）：需用户预先订阅
- **短信**（备用，Phase 2）：urgent 级别时使用

### 9.3 防误报

- 仅在子女发送了内容后才检测已读（不发不查）
- 老人可设置"免打扰"时段（如 22:00-08:00）
- 子女可手动标记"已确认安好"关闭告警

## 10. 前端页面规划

### 10.1 子女端页面

```
pages/
├── index/                    # 首页（老人状态总览 + 快速发送入口）
├── send/                     # 发送牵挂内容
│   ├── index.vue             # 编辑发送页
│   └── ai-suggest.vue        # AI 生成建议页
├── history/                  # 发送历史（含已读状态）
├── elder/                    # 老人详情
│   ├── status.vue            # 老人状态详情 + 活跃度日历
│   └── settings.vue          # 告警设置 + 社区联系人管理
├── bind/                     # 绑定管理
│   ├── invite.vue            # 生成邀请码
│   └── list.vue              # 绑定关系列表
├── alerts/                   # 告警中心
└── profile/                  # 个人设置
```

### 10.2 老人端页面

```
pages/
├── view/                     # 查看牵挂内容（极简大字体）
│   └── [id].vue              # 内容详情页（从微信卡片直接进入）
└── elder-home/               # 老人首页（汇聚所有收到的内容，P1）
```

### 10.3 老人端设计原则

- **字体**：正文 ≥ 36rpx，标题 ≥ 44rpx
- **按钮**：最小触控区域 96rpx × 96rpx
- **颜色**：高对比度，主色调温暖（暖橙/暖黄）
- **布局**：单列，内容上下排布，无左右滑动
- **交互**：无需打字，预设表情一键回应

## 15. 核心交互流程

### 15.1 首次使用流程

```
子女首次打开小程序
  → 微信静默登录
  → 选择身份："我是子女" / "我是长辈"
  → (子女) 设置昵称 → 生成邀请码/二维码
  → 帮老人扫码绑定 或 发送邀请码给老人
  → 绑定成功 → 进入首页
```

### 15.2 日常使用流程

```
子女打开小程序
  → 首页看到老人状态（今日已读✓ / 今日未读）
  → 点击"发送牵挂" → 拍照/选图 + 输入文字
    或 → 点击"AI帮我写" → 选择建议内容 → 确认
  → 发送 → 生成小程序卡片
  → 转发卡片给老人（微信聊天）
  → 等待老人查看 → 状态变为"已读"

老人收到微信消息
  → 点击小程序卡片
  → 看到孩子分享的温暖内容（大字体、大图片）
  → (可选) 点击表情回应
  → 关闭 → 系统已记录"已读"
```

## 16. 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 老人不会点微信卡片 | 核心流程断裂 | 首次绑定时子女现场教一次；卡片封面设计醒目，标题写"XX给您的牵挂" |
| 小程序订阅消息限制 | 无法主动推送告警 | 引导子女预先订阅多次；urgent 级别走短信备用 |
| 老人换手机/重装微信 | openid 不变不影响 | openid 与微信账号绑定，换手机登录同一微信号即可 |
| AI 生成不当内容 | 用户体验差 | 安全过滤 + 子女确认后才发送 |
| 子女使用频率下降 | 系统失去价值 | AI 定时提醒子女发送；每周推送"牵挂周报" |
